Abstract

RecoMart requires a maintainable data pipeline that can ingest customer behaviour data, validate data quality, engineer recommendation-ready features, and train reusable recommendation models. The implemented repository addresses this requirement through a modular batch workflow that combines synthetic source generation, API-driven product collection, timestamped raw storage, automated validation, structured preparation, feature-store registration, recommendation-model training, and orchestration scripts.

The solution is intentionally organised into independent stages so each step can be executed, tested, and monitored separately. Clickstream and purchase-history datasets are generated locally for repeatable experimentation, while product metadata is fetched from the Fake Store API to simulate an external catalogue dependency. Each dataset is ingested into source-specific raw zones, validated for completeness and basic business rules, merged into a prepared analytical dataset, and transformed into user, product, and interaction feature tables.

These feature tables are persisted to a SQLite-backed feature store and converted into a training dataset for collaborative recommendation modelling. The training pipeline builds a user-item interaction matrix and fits a K-nearest-neighbours model using cosine similarity. Evaluation utilities estimate ranking performance with holdout logic and compare alternative recommenders — popularity, user-based, item-based, and hybrid variants.

The project establishes a credible end-to-end baseline for recommendation-system data management, while also surfacing concrete improvement opportunities in experiment tracking, online serving, richer feature versioning, and stronger product-ranking inference interfaces — detailed in Section 17, Future Scope.

Problem Statement

Why RecoMart needs a managed data pipeline, not just a model
Section 02

RecoMart is an e-commerce startup that wants to increase customer engagement, improve product discovery, and drive cross-selling by recommending relevant products to users. The business challenge is not only to build a recommendation model, but to establish a dependable data pipeline that keeps recommendation inputs fresh, validated, traceable, and ready for downstream machine-learning use.

Central engineering question: how can RecoMart continuously transform heterogeneous behavioural and catalogue data into reliable machine-learning artifacts that support recommendation experiments and future production deployment?

Objectives

  • Automate ingestion for user clickstream data, purchase history, and product metadata.
  • Store raw data in a structured layout with source-level separation and timestamped files.
  • Validate incoming data for missing values, duplicates, invalid events, and invalid numeric values.
  • Prepare a clean, integrated dataset suitable for transformation and analysis.
  • Engineer recommendation-oriented user, product, and interaction features.
  • Persist features in a reusable feature store with metadata registration and version labels.
  • Build a collaborative-filtering training dataset and a reusable trained model artifact.
  • Evaluate the recommender with Precision@K, Recall@K, and NDCG@K; orchestrate the full workflow end to end.

Business Context

Recommendation systems are valuable because they influence both user experience and revenue. In the RecoMart scenario, personalisation reduces browsing friction, increases the likelihood of repeat sessions, and improves the visibility of long-tail products that might otherwise stay hidden behind category pages. From a data-platform perspective, this means the organisation needs an architecture that turns raw activity logs into business-ready insight and model-ready features with minimal manual intervention.

The codebase reflects this context by prioritising modularity and reproducibility: ingestion, validation, transformation, feature management, training, and orchestration are separated into distinct modules so that operational ownership can eventually be distributed across the data platform team.

Scope note: this assignment implements the pipeline against synthetic and API-sourced data at a scale appropriate for coursework (thousands, not millions, of rows). The architecture and module boundaries are nonetheless designed to generalise to production data volumes — see Section 05, System Architecture.

05System Architecture
05

System Architecture

A ten-stage layered batch pipeline, each stage producing a stable handoff artifact
Section 05

The RecoMart architecture follows a staged batch design in which each module reads a stable input and writes a stable output for the next stage. This keeps the workflow inspectable, independently restartable, and far easier to debug than a monolithic script — if product ingestion fails, only Stage 2 needs to be re-run.

Architectural Interpretation

The architecture intentionally separates storage concerns from modelling concerns. Raw data is preserved first, then cleaned into a prepared dataset, then transformed into feature tables, and only then turned into model inputs. This progression supports auditability: each layer has a clear responsibility and a stable artifact path, matching the lineage documented in Section 12.

From a maintainability standpoint, the strongest aspect of the architecture is that most modules can be re-run independently. If product ingestion fails, raw data can be re-fetched without retraining. If feature logic changes, the team can regenerate the feature store and training data without rebuilding the source generator — isolating failure domains the same way production data platforms do.

06Methodology
06

Methodology

A disciplined, six-step progression from collection to model artifact
Section 06

The project uses a batch-oriented recommendation-data methodology centred on modular transformation layers. Rather than jumping directly from source files to model training, the repository adopts a disciplined progression from collection to validation to prepared data to feature store to model artifact. This reduces hidden coupling and makes the end-to-end pipeline easier to explain, test, and extend.

1

Source creation & acquisition

Behavioural CSVs are generated locally, while product metadata is fetched from an external API. This creates both reproducibility (deterministic seeds) and realistic external-integration complexity (network failures, schema shape).

2

Landing & preservation

All inbound data is copied or saved into raw source folders with timestamps, so the original payloads are preserved before any mutation — the foundation for replay and audit.

3

Quality enforcement

Validation scripts inspect row counts, duplicates, missing values, invalid events, invalid prices, and invalid quantities before downstream transformation is allowed to proceed.

4

Preparation & integration

Purchase history is joined with product metadata and then linked with clickstream behaviour, so each analytical row carries both transaction and product context.

5

Feature generation

User-level, product-level, and interaction-level features are created and written into the SQLite feature store for reuse across training and future inference.

6

Modelling & evaluation

Interaction scores are transformed into rating-style training records that support collaborative recommendation experiments and holdout-based ranking evaluation.

Methodological Strengths

  • Traceability across layers via explicit artifact paths and versioned directories.
  • Supports both educational clarity and future production hardening.
  • Evaluation is separated from training, making experimentation easier.
  • Structure is simple enough to debug with logs and local files alone.

Data Collection & Raw Storage

Source-specific ingestion into a timestamped, partitioned landing zone
Section 07

Data collection combines deterministic local generation with external retrieval. The generator creates 10,000 clickstream events and 2,500 purchase records spanning a 90-day window, while the product-ingestion module requests 20 products from the Fake Store API — simulating a realistic scenario where internal user behaviour must be joined with external product content.

The ingestion layer is intentionally source-specific: CSV datasets are copied into raw storage using separate scripts for clickstream and purchase history, while product ingestion uses an HTTP client with retry behaviour. This separation is good engineering practice, since different source types typically need different failure handling and monitoring logic.

Raw Storage Layout

data/ ├── source/ # deterministic generator output │ ├── clickstream.csv │ └── purchase_history.csv └── raw/ # timestamped, source-partitioned landing zone ├── clickstream/ │ └── clickstream_20260718_105138.csv ├── purchase/ │ └── purchase_20260718_105138.csv └── products/ └── products_20260718_105140.json

RReliability Features

  • CSV ingestion validates source-file existence before copy.
  • Copy operations retry up to three times with delay.
  • API ingestion retries up to three times with fixed wait intervals.
  • Every ingestion event is logged to logs/ingestion.log.

SStorage Assessment

The layout is simple but effective for a course-scale project: it preserves raw payloads, supports re-runs, and is compatible with future migration to cloud object storage (S3 / GCS) if RecoMart later needs larger-scale historical retention.

09Data Preparation
09

Data Preparation

Merging three sources into one clean analytical table
Section 09

The preparation stage loads the latest raw files, flattens the nested product-rating object, merges purchase records with product metadata, and joins clickstream events using user_id and product_id. After merging, the pipeline removes duplicate rows and duplicate columns, imputes missing event labels and timestamps, standardises category text, converts timestamp fields to proper datetime types, and creates derived fields such as purchase_amount, purchase_year, purchase_month, and purchase_day.

1Before — Raw Inputs

  • 3 disconnected files, 3 different shapes
  • Nested JSON rating object
  • No shared derived time fields
  • Purchase and click events unlinked

2After — Prepared Table

  • 1 integrated table, 3,405 rows × 16 columns
  • Flat rating_rate / rating_count fields
  • Derived purchase_year/month/day
  • Purchase rows enriched with click-behaviour context

The resulting prepared dataset is written to data/processed/prepared_dataset.csv. In the current repository snapshot, this artifact contains 3,405 rows and 16 columns. This stage is especially important because it transforms isolated source records into a consistent analytical layer that supports both feature engineering (Section 10) and exploratory business analysis.

Validation protects the pipeline from obvious data defects, while preparation makes the data structurally useful. Treating these as separate stages — rather than one combined script — is one of the more mature design choices in the project.

The project uses DVC to version the major data layers rather than relying only on Git. Three DVC-tracked outputs are present in the repository: data/raw.dvc, data/processed.dvc, and database.dvc. For a student project, this is a strong foundation. DVC is used in the correct conceptual place — not to replace Git, but to complement it by tracking heavier data artifacts and generated outputs. The next maturity step would be to add explicit provenance fields (source ingestion timestamp, generator seed, feature-run ID, evaluation-run ID) so end-to-end reproducibility becomes tighter still.

13Model Training
13

Model Training

User-user collaborative filtering via cosine-similarity KNN
Section 13

Model training starts with models/prepare_training_data.py, which reads the interaction_features table from the SQLite feature store and converts interaction_score into a rating-style field. Duplicate user-product records are removed before the dataset is saved to data/model/training_data.csv. In the current snapshot, the resulting training artifact contains 2,648 rows and 3 columns.

The training script then pivots the dataset into a user-item matrix where rows represent users, columns represent products, and cell values represent interaction scores. The current matrix contains 497 users and 20 products. A cosine-distance NearestNeighbors model is trained on this matrix using the brute-force algorithm and a neighbour count of 10.

MTraining Artifact

The serialised model is saved to models/saved_models/knn_model.pkl with both the fitted estimator and the user-item matrix. Persisting the matrix alongside the model matters because recommendation lookup depends on the training-space indices.

DCurrent Modelling Design

The active model is a user-user collaborative-filtering core. The repository's inference helper primarily exposes similar-user outputs, while evaluation utilities convert neighbour signals into ranked item recommendations during offline testing.

Why KNN Was a Sensible Baseline

  • Simple to explain and fast to prototype against a small, sparse interaction matrix.
  • Works directly from implicit feedback transformed into interaction scores.
  • Allows easy comparison with popularity and item-based alternatives (Section 14).
  • Creates a useful baseline before moving to matrix factorisation or neural recommenders.

The most important evaluation insight is not that KNN "works," but that the item-based formulation outperforms the currently serialised user-based baseline across all three metrics. This gives the team a clear, low-risk next optimisation target for Section 17.

Metric Definitions

Precision@K

Relevant items in top-K ÷ K. Measures how much of what was shown to the user was actually useful.

Recall@K

Relevant items in top-K ÷ total relevant items. Measures how much of the useful catalogue was surfaced.

NDCG@K

Rewards relevant items appearing earlier in the ranked list, using a logarithmic position discount.

With a single held-out relevant item per user (the leave-one-out protocol used here), Recall@K reduces to a hit/no-hit indicator, which is why Recall@10 tracks closely with Precision@10 × 10 in the table above.

Pipeline Orchestration

Eleven-task DAG sequencing generation through evaluation
Section 15

End-to-end execution is managed through two orchestration entry points: a direct Python pipeline script and a Prefect-compatible pipeline module. Both sequence the same major stages: source generation, ingestion, validation, preparation, feature engineering, feature registration, training-data creation, model training, and evaluation. The Prefect-oriented variant extends this with additional metric computation and recommendation generation.

Generate Source Data

      ├─▶ Validate Clickstream ─┐

Conclusion & Future Scope

A credible baseline with clearly scoped next steps
Section 17

The RecoMart project successfully implements the core layers of an end-to-end data management pipeline for a recommendation system. It ingests multi-source data, preserves raw copies, validates quality, prepares integrated analytical data, engineers recommendation features, stores those features in SQLite, versions major data artifacts through DVC, trains a collaborative-filtering baseline, and orchestrates the sequence in modular scripts. From an assignment perspective, the repository satisfies the essential expectations for data engineering structure, reproducibility, and model-readiness.

Just as importantly, the project surfaces realistic next-step opportunities. Evaluation shows that item-based KNN outperforms the currently saved user-based baseline, indicating the system already contains enough signal to justify another round of model refinement. The codebase therefore does more than satisfy a checklist — it creates a usable foundation for iterative improvement.

Future Scope

MModelling

  • Promote item-based KNN, matrix factorisation, or hybrid ranking models into the primary training path.
  • Expose direct product-ranking inference instead of similar-user output alone.
  • Add explicit MLflow logging to the training and evaluation scripts.

PPlatform

  • Upgrade the local SQLite feature store to Feast or another point-in-time-aware feature platform.
  • Support streaming or micro-batch ingestion for fresher behaviour updates.
  • Introduce automated tests, CI, schema contracts, and stronger metadata-deduplication rules.

DDemonstration

Deploy a lightweight API or dashboard so recommendations can be demonstrated interactively — this would also give the team a natural place to add live monitoring for data drift and model staleness.

Final Assessment

As of 18 July 2026, RecoMart has a credible and extensible recommendation-data pipeline with clear evidence of engineering thoughtfulness. The biggest remaining step is not building the pipeline from scratch, but hardening and productising the strong baseline that already exists.

Appendix C — Risk Register & Assumptions

Known limitations, explicitly stated rather than discovered by a reviewer
Appendix

Documenting known limitations up front is part of responsible data-platform engineering. The table below lists the risks the team is aware of in the current implementation, alongside the assumption or mitigation in place and the priority for the next iteration.

#Risk / LimitationCurrent MitigationPriority
1Synthetic clickstream/purchase data may not reflect real user behaviour distributions.Faker seeds and event-weight ratios were chosen to approximate plausible e-commerce patterns.Medium
2Product catalogue is capped at 20 SKUs (Fake Store API limit).Sufficient for demonstrating the pipeline; noted as a scale constraint, not a design flaw.Low
3External API calls can time out or rate-limit.Tenacity retry (3 attempts) around ingest_products.py; failures are logged, not silently swallowed.Handled
4SQLite feature store is single-file and local — not safe for concurrent writers.Acceptable for a single-pipeline, single-machine coursework context.Medium
5MLflow directory exists but training does not yet log to it explicitly.Flagged transparently in Sections 04 and 17 rather than overstated in this report.Medium
6KNN evaluation metrics can shift run-to-run since source data is regenerated each time.Report states the exact snapshot date (18 July 2026) for every reported number.Low
7No automated tests or CI currently guard the pipeline against regressions.Manual end-to-end run before each report update; listed explicitly under Future Scope.High
8Feature metadata registration can insert duplicate rows on repeated pipeline runs.Not currently deduplicated; acceptable for demonstration, tracked as a cleanup item.Medium
9Cold-start users/products (no interaction history) are not explicitly handled by the primary KNN path.The hybrid variant in Section 14 partially addresses this; not yet the default model.Medium

Working Assumptions

  • Interaction-score weighting (view=1 … purchase=5) is treated as a reasonable proxy for intent, not a calibrated business metric — it has not been validated against real conversion data.
  • The 80/20-style holdout used in evaluation assumes each evaluable user has enough history to support a meaningful held-out item; users with a single interaction are excluded from the 476 evaluable users cited in Section 14.
  • Cosine similarity on raw interaction scores assumes scale-comparability across users; no per-user normalisation is currently applied before fitting the KNN model.

Definitions for terms used throughout this report
Appendix
TermDefinition
ClickstreamThe sequence of user interaction events (view, click, wishlist, add-to-cart) captured during a browsing session.
Collaborative FilteringA recommendation approach that predicts a user's preferences based on the preferences of similar users (user-based) or similar items (item-based).
Cosine SimilarityA measure of similarity between two vectors based on the cosine of the angle between them; used here to compare user interaction profiles.
DVC (Data Version Control)A Git-compatible tool for versioning large data files and directories without storing their full content in Git itself.
Feature StoreA centralised system for storing, documenting, and serving engineered features consistently to both training and inference workloads.
Holdout EvaluationAn evaluation protocol where part of a user's history is deliberately withheld from training and used as ground truth for testing.
Interaction ScoreA single numeric value representing the strength of a user's engagement with a product, derived by weighting event types.
KNN (K-Nearest Neighbours)A non-parametric algorithm that makes predictions based on the K most similar data points to a given query point.
NDCG (Normalised Discounted Cumulative Gain)A ranking-quality metric that rewards placing relevant items near the top of a ranked recommendation list.
Precision@KThe proportion of the top-K recommended items that are actually relevant to the user.
Raw Zone / Landing ZoneStorage location where ingested data is preserved in its original, unmodified form before any cleaning or transformation.
Recall@KThe proportion of all relevant items that are successfully captured within the top-K recommendations.
User-Item MatrixA matrix where rows represent users, columns represent items, and cell values represent an interaction signal (e.g. interaction score).
Backoff / RetryA resilience pattern where a failed operation is automatically retried, typically with an increasing delay between attempts.
Data LineageA traceable record of where a piece of data originated and every transformation it passed through before reaching its current form.
Orchestration DAGDirected Acyclic Graph — a task dependency graph with no cycles, used to schedule and sequence pipeline stages.
Synthetic DataArtificially generated data designed to resemble real data statistically, used here for reproducible local experimentation.

E.3 — Interaction-Score Weighting (transformation/interaction_features.py)

EVENT_SCORE = {"view": 1, "click": 2, "wishlist": 3, "add_to_cart": 4, "purchase": 5}

def create_interaction_features(df):
    interaction = df[["user_id", "product_id", "event"]].copy()
    interaction["interaction_score"] = interaction["event"].map(EVENT_SCORE).fillna(0)
    return interaction
